# Webpack5(part1)

版本号：

webpack: 5.72.0 webpack-cli: 4.9.2

# 认识webpack

随着前端的快速发展，前端开发已经变得越来越复杂了：

  • 需要通过模块化的方式来开发；

  • 需要使用一些高级的特性来加快我们的开发效率或者安全性，比如通过ES6+、TypeScript开发脚本逻辑，通过sass、less等方式来编写css样式代码；

  • 需要对代码进行压缩、合并以及其他相关的优化；

  • 等等……

目前在前端开发中我们通常都会直接使用三大框架来实现上述各种开发需求： Vue、React、Angular ；

这三大框架的创建过程我们都是借助于脚手架(CLI)的；

事实上，三大框架对应的脚手架 Vue-CLI、create-react-app、Angular-CLI 都是 基于webpack 来帮助我们支持模块化、less、TypeScript、打包优化等的。

# webpack到底是什么

官方解释：

webpack is a static module bundler for modern JavaScript applications.

webpack是一个用于现代JavaScript应用程序的静态模块打包工具。

我们来对上面的解释进行拆解：

  • 打包bundler : webpack可以帮助我们进行打包，所以它是一个打包工具
  • 静态的static : 这样表述的原因是我们最终可以将代码打包成最终的静态资源(部署到静态服务器)
  • 模块化module : webpack默认支持各种模块化开发，ES Module、CommonJS、AMD等
  • 现代的modern : 因为现代前端开发越来越复杂，产生了各种各样的需求，所以催生了webpack的出现和发展

# webpack安装和使用

使用webpack时，项目名称推荐使用英文，尽量避免出现中文字符。

webpack官方的图片 ：

webpack的官方文档 (opens new window)

webpack的官方中文文档 (opens new window)

documentation : 文档详情，说明书。

webpack的运行是依赖 Node环境 的，所以我们电脑上必须有Node环境：

  • Node官网 (nodejs.org) (opens new window)
  • Node的下载十分简单，进入官网后就能看到两个下载按钮
  • 它们分别代表LTS(稳定版本)和Current(最新版本)
  • 选择其中一个下载即可，一般推荐稳定版本
  • 下载完成后，点击安装包，之后一路next就能成功安装
  • 安装包在安装node.js时会同时安装npm

webpack的安装目前分为两个： webpack、webpack-cli

它们之间的关系：

  • 执行webpack命令，会执行node_modules下的.bin目录下的webpack；
  • webpack在执行时是依赖webpack-cli的，如果没有安装就会报错；
  • 而webpack-cli中代码执行时，才是真正利用webpack进行编译和打包的过程；
  • 所以在安装webpack时，我们需要同时安装webpack-cli(第三方的脚手架事实上是没有使用webpack-cli的，而是类似于自己的vue-service-cli的东西)

npm install webpack webpack-cli –g # 全局安装

npm install webpack webpack-cli –D # 局部安装

这是我安装的版本：

webpack的默认打包：

我们可以通过webpack进行打包，之后运行打包之后的代码

  • 在目录下直接执行 webpack 命令

    webpack

生成一个dist文件夹，里面存放一个main.js的文件，就是我们打包之后的文件：

  • 这个文件中的代码被压缩和丑化了；
  • 另外我们发现代码中依然会存在ES6的语法，比如箭头函数、const等，这是因为默认情况下webpack并不清楚我们打包后的文件是否需要转成ES5之前的语法，后续我们需要通过babel来进行转换和设置；

我们发现是可以正常进行打包的，但是有一个问题，webpack是如何确定我们的入口的呢？

  • 事实上，当我们运行webpack时，webpack默认会查找当前目录下的 src/index.js 作为入口；
  • 所以，如果当前项目中没有存在 src/index.js 文件，那么会报错；

当然，我们也可以通过配置来指定入口和出口：

npx webpack --entry ./src/main.js --output-path ./build

注意： 浏览器无法识别模块化、TypeScript、less、sass等资源，而webpack可以将这些资源打包成浏览器可以识别的 JS 和 css 等，正如前面放置的那张webpack官方的图片所显示的一样。

创建局部的webpack：

前面我们直接执行webpack命令使用的是全局的webpack，全局webpack一般是用作测试，我们在开发中通常使用的是局部的webpack，让每个单独的项目都拥有一个属于它自己的webpack打包工具，使用局部的webpack可以按照下面的步骤来操作：

第一步：创建package.json文件，用户管理项目的信息、库依赖等

  • npm init

  • 在编译器终端执行上方命令后，便可以开始逐步填写项目的相关信息

  • 如果只是简单测试，可以全部使用默认信息，一路回车即可

  • 也可以执行下方命令，直接创建默认文件

  • npm init -y

第二步：安装局部的webpack

  • 命令一：

    npm install webpack webpack-cli --save-dev

  • 命令二：

    npm install webpack webpack-cli -D

  • 上述两个命令的后缀，意思是一样的，都表示将安装内容放入项目的开发环境中；

  • 开发环境：里面存放的都是工具类package，负责代码的各种优化工作，不会出现在最终发送至服务器的静态资源里

第三步：使用局部的webpack进行打包

  • 通常，直接在终端输入 webpack 的话，使用的都是全局webpack；

  • 想要使用局部的webpack，需要输入另外的命令；

  • 命令一：

    node_modules/.bin/webpack

  • 命令二：

    在package.json文件中，修改script如下

    "scripts": {

    ​ "build": "webpack"

    }

    然后便可以直接在终端这样输入命令：

    npm run build

    在script内做修改，虽然属性"build"对应的内容只有"webpack"，但脚本会自动去目录 node_modules/.bin 下寻找并调用 webpack 文件

  • 命令三：

    npx webpack

    同样，npx命令会自动去目录 node_modules/.bin 下寻找并调用 webpack 文件

# webpack的配置文件

在通常情况下，webpack需要打包的项目是非常复杂的，并且我们需要一系列的配置来满足要求，默认配置必然是不可以的。

我们可以在项目根目录下创建一个 webpack.config.js 文件(默认名称)，来作为webpack的配置文件。

可以在配置文件中修改webpack的默认出入口：

// webpack.config.js
const path = require("path")
// 导出配置信息
module.exports = {
  // 修改webpack的入口和出口文件的路径和名称
  // 入口文件
  entry: "./src/main.js",
  // 出口文件
  output: {
    path: path.resolve(__dirname, "./build"),
    // 将导出的bundle.js文件放入输出文件夹build目录下的js文件夹内
    filename: "js/bundle.js"
  }
}

webpack的依赖图：

webpack到底是如何对我们的项目进行打包的呢？

  • 事实上，webpack在处理应用程序时，它会根据命令或者配置文件找到入口文件；
  • 从入口开始，会生成一个 依赖关系图 ，这个 依赖关系图 会包含应用程序中 所需的所有模块 (比如入口的引用文件，引用文件的引用文件等等，不断延伸)；
  • 然后遍历图结构，打包一个个模块(根据文件格式的不同使用不同的 loader 来解析)；

终端报错。

Module parse failed: Unexpected token (1:0)

You may need an appropriate loader to handle this file type

上面的错误信息告诉我们需要一个 loader 来加载这个css文件，但是 loader 是什么呢？

  • loader 可以用于对 模块的源代码 进行转换；
  • 我们可以将css文件也看成是一个模块，我们是通过import来加载这个模块的；
  • 在加载这个模块时，webpack其实不知道如何对其进行加载，我们必须制定对应的loader来完成这个功能；

那么，我们需要一个怎样的loader呢？

  • 对于加载css文件来说，我们需要一个可以读取css文件的loader；
  • 最常用的是 css-loader ；

css-loader的安装：

npm install css-loader -D

css-loader的使用方案：

如何使用这个loader来加载css文件呢？有三种方式：

  • 内联方式；
  • CLI方式(webpack5中不再使用)；
  • 配置方式；

内联方式 ：内联方式使用较少，因为不方便管理

  • 在引入的样式前加上使用的loader，并且用符号 ! 分割

    // element.js
import "css-loader!../css/style.css"
    1
    2

CLI方式：

  • 在webpack5的文档中已经没有了 --module-bind ;
  • 实际应用中也比较少使用，因为不方便管理。

loader配置方式： 配置方式表示的意思是在我们的webpack.config.js文件中写明配置信息

  • module.rules中允许我们配置多个loader(因为我们也会继续使用其他的loader，来完成其他文件的加载)；
  • 这种方式可以更好的表示loader的配置，也方便后期的维护，同时也让你对各个Loader有一个全局的概览；

module.rules的配置如下：

rules属性对应的值是一个数组 : [Rule]

数组中存放的是一个个的Rule，Rule是一个对象，对象中可以设置多个属性：

  • test属性 : 用于对 resource(资源)进行匹配的，通常会设置成正则表达式；

  • use属性 : 对应的值是一个数组 : [UseEntry]

UseEntry是一个对象，可以通过对象的属性来设置一些其他属性

  • loader : 必须有一个loader属性，对应的值是一个字符串
  • options : 可选的属性，值是一个字符串或者对象，值会被传入到loader中
  • query : 目前已经使用options来替代

loader属性是use属性的一种简写

loader: "css-loader" // 语法糖
use: [
  { loader: "css-loader" }
] // 原来的语法

    rules: [

          "style-loader",
          "css-loader"

      }
    ]
  }
}

# PostCSS工具

什么是PostCSS呢？

  • PostCSS是一个通过JavaScript来转换样式的工具；
  • 这个工具可以帮助我们进行一些CSS的转换和适配，比如自动添加浏览器前缀、css样式的重置；
  • 但是实现这些功能，我们需要借助于PostCSS对应的插件；

如何使用PostCSS呢？主要就是两个步骤：

  1. 查找PostCSS在构建工具中的扩展，比如webpack中的postcss-loader；
  2. 选择添加你需要的PostCSS相关的插件；

在使用webpack打包的开发中我们会选择使用postcss-loader

npm install postcss-loader -D

注意：因为postcss需要有对应的插件才会起效果，所以我们需要配置它的plugin，

可用的plugin有autoprefixer、postcss-preset-env；

通常使用postcss-preset-env :

  • postcss-preset-env是一个postcss的插件；
  • 它可以帮助我们将一些现代的CSS特性，转成大多数浏览器认识的CSS，并且会根据目标浏览器或者运行时环境添加所需的polyfill；
  • 也包括会自动帮助我们添加autoprefixer(所以相当于已经内置了autoprefixer)

npm install postcss-preset-env -D

下载安装完如何使用？

我们可以在项目根目录下创建一个 postcss.config.js 文件，这是PostCSS的配置文件，代码如下：

// postcss.config.js
module.exports = {
  plugins: [
    require("postcss-preset-env")
  ]
}

# webpack的图片资源打包

在项目开发中，我们经常会在页面中插入图片，比较常见的使用图片的方式有两种：

  • img元素，设置src属性
  • 其他元素(比如div)，设置background-image的css属性

在webpack打包图片的过程中，同样需要使用相应的工具来对图片文件进行解析，在 webpack5 之前通常是使用 file-loader 和 url-loader 。

# file-loader写法

先下载安装file-loader：

npm install file-loader -D

file-loader的作用就是帮助我们处理 import/require() 方式引入的文件资源，并且将它放到我们 输出的文件夹 中；

配置处理图片的Rule：

{
  test: /\.(jpe?g|png|gif|svg)$/,
  use: {
    loader: "file-loader"
  }
}

# 文件的命名规则

在项目开发中，我们通常希望处理后的文件名称按照一定的规则显示，比如在输出文件夹dist内，再创建一个image文件夹然后将图片放入其中，或是创建一个和图片原来所在的文件夹所同名的文件夹，再放入图片，同时保留图片原来的名称和扩展名，为了防止重复，也可以为其添加一个hasn值。

这个时候我们可以使用 PlaceHolders 来完成，webpack给我们提供了大量的PlaceHolders来显示不同的内容：

  • file-loader|webpack (opens new window)
  • 我们可以在上面文档中查阅自己需要的placeholder

下面介绍几个常用的placeholder：

  • [ext] : 保留原文件的扩展名
  • [name] : 保留原文件的名称
  • [hash] : 使用MD4的散列函数处理，生成的一个128位的hash值(32个十六进制)
  • [folder] : 保留原文件的文件夹名称
  • [contentHash] : 在file-loader中和[hash]结果是一致的(在webpack的一些其他地方不一样，后面会讲到)
  • [hash:<length>] : 定制hash的长度，默认32个字符太长了
  • [path] : 文件相对于webpack配置文件的路径

现在使用placeholder设置文件的名称，代码如下：

      // 在outputPath使用placeholder无效
      outputPath: "img",
      name: "[name]_[hash:6].[ext]"
    }
  }
},

# url-loader写法

url-loader和file-loader的工作方式是相似的，但是可以将较小的文件，转成 base64的URI 。

下载安装url-loader：

npm install url-loader -D

在配置文件中，file-loader和url-loader写一个就够了

将前面的file-loader代码删除，写入url-loader代码，如下：

{
  test: /\.(jpe?g|png|gif|svg)$/,
  use: {
    loader: "url-loader",
    options: {
      // outputPath: "img",
      name: "[folder]/[name]_[hash:6].[ext]"
    }
  }
},

默认情况下，url-loader会将所有图片都转为base64编码并嵌入对应的入口文件中，所以输出文件夹里不会有图片，这样做可以减少请求次数，

譬如现在我们有50张2kb的小图片，如果分别放入对应文件夹的话，浏览器需要前后发送50次网络请求，来将这50张小图片下载下来，既消耗了浏览器性能，又加重了服务器压力，

如果将这50张小图片转成base64编码并嵌入一份js代码文件中，只需要一次请求，就可以随js文件一起被下载至浏览器，大大减轻了服务器的压力，

但是 ，如果图片过大，将其转成base64编码之后，浏览器下载解析它所嵌入的js文件时就会变得非常慢，甚至可能出现短暂的白屏等其他影响用户体验的现象，所以，一般建议只对小图片进行base64的转换，

那么如何设置只针对小图片的base64转换呢？

使用 limit属性 。代码如下：

{
  test: /\.(jpe?g|png|gif|svg)$/,
  use: {
    loader: "url-loader",
    options: {
      name: "[folder]/[name]_[hash:6].[ext]",
      // 将小于100kb的图片转为base64
      limit: 100 * 1024
    }
  }
},

# webpack5写法

在webpack5之前，加载资源我们需要使用一些loader，比如上述示例。

在webpack5之后，我们可以直接使用 资源模块类型(asset module type) , 来替代上面的这些loader。

资源模块类型(asset module type)，通过添加4中新的模块类型，来替换所有这些loader：

  • asset/resource 发送一个单独的文件并导出URL。之前通过使用 file-loader 实现；
  • asset/inline 导出一个资源的 data URI。之前通过使用 url-loader 实现；
  • asset/source 导出资源的源代码。之前通过 raw-loader 实现；
  • asset 在导出一个 data URI 和发送一个单独的文件之间自动选择。之前通过使用 url-loader ，并且配置资源体积限制实现；

比如加载图片，默认写法如下：

{
  test: /\.(jpe?g|png|gif|svg)$/,
  type: "asset/resource",    
}

# webpack的plugin

webpack的另一个核心是Plugin，官方有这样一段对Plugin的描述：

While loaders are used to transform certain types of modules, plugins can be leveraged to perform a wider range of tasks like bundle optimization, asset management and injection of environment variables.

加载器用于转换某些类型的模块，而插件可以用于执行更广泛的任务，如包优化、资源管理和环境变量注入。

# CopyWebpackPlugin

在前面我们将在网上下载的favicon.ico文件与模板文件一起放进了public文件夹，如果我们希望在webpack打包的时候直接将favicon.ico文件复制到dist文件夹中，或者说，将public文件夹目录下除了index.html以外的文件都复制到dist文件夹中的话，要如何实现这个功能呢？

  • 可以使用插件 CopyWebpackPlugin 来实现。

下载安装：

npm install copy-webpack-plugin -D

CopyWebpackPlugin的配置规则如下：

  • 复制的规则在patterns中设置；
  • from：设置从哪一个源中开始复制；
  • to：复制到的位置，可以省略，会默认复制到打包的目录下；
  • globOptions：设置一些额外的选项，其中可以编写需要忽略的文件：
    • index.html：不需要复制，因为我们已经通过HtmlWebpackPlugin完成了index.html的生成；

配置文件：

// webpack.config.js
const { CleanWebpackPlugin } = require("clean-webpack-plugin")
const HtmlWebpackPlugin = require("html-webpack-plugin")
const { DefinePlugin } = require("webpack")
const CopyWebpackPlugin = require("copy-webpack-plugin")

module.exports = {
  module: {
    rules: [...]
  },
  plugins: [
    new CleanWebpackPlugin(),
    new HtmlWebpackPlugin({
      template: "./public/index.html",
      title: "webpack_plugins"
    }),
    new DefinePlugin({
      BASE_URL: "'./'"
    }),
    // 配置复制文件和路径
    new CopyWebpackPlugin({
      patterns: [
        {
          from: "public",
          to: "./",
          globOptions: {
            // 忽略、不复制的文件
            ignore: [
              "**/index.html"
            ]
          }
        }
      ]
    })    
  ]          
}

# webpack的mode

Mode配置选项，可以告知webpack使用响应模式的内置优化：

  • 默认值是production(什么都不设置的情况下)；
  • 可选值有： 'none' | 'development' | 'production' ;

在前面的代码运行过程中，如果webpack编译出错，它所返回的报错信息的代码行数提示通常都是：

main.js : 1

这是因为webpack对我们的代码进行了压缩和丑化，

所有代码都在一行，并且变量名称等都被做了一定程度的修改，

所以我们往往看不懂webpack打包后的文件，

报错返回的代码行数提示变得毫无意义，

当项目的代码量巨大时，报错返回的报错原因也将很难派上用场，因为找不到是哪里的问题。

综上种种问题，webpack为了方便开发者进行测试，提供了 mode配置 。

在开发阶段，我们会给 mode 设置为 development (开发)模式，然后在给 devtool 设置为 source-map ，

配置如下：

// webpack.config.js
module.exports = {
  // 设置模式
  // development 开发阶段，会设置development
  // production 准备打包上线的时候，设置production
  mode: "development",
  // 设置source-map, 建立js映射文件，方便调试代码和错误
  devtool: "source-map",
  ......
}

执行webpack打包后，会在dist文件夹里生成一个与出口js文件main.js对应的main.js.map文件，这里面存放的是打包之后的文件和未打包的文件它们之间的 映射关系 ，

作用是什么？

有了这份映射关系，之后在开发过程中如果出现了报错，报错信息则会恢复 "正常"，能够指出报错原因，精准地提示 原项目(被打包之前) 对应的错误文件和错误行数，让开发者可以根据提示直接去对应文件中的对应行数修改错误，而不用再盯着打包文件的 那一行看不懂的压缩代码 发愁了。

在准备部署项目的时候，我们会将 mode 设置为 production (生产)模式。

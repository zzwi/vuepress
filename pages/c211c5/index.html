包管理工具详解

# 25.包管理工具详解

# 代码共享方案

我们已经学习了在JavaScript中可以通过模块化的方式将代码划分成一个个小的结构：

  • 在以后的开发中我们就可以通过模块化的方式来封装自己的代码，并且 封装成一个工具 ；
  • 这个工具我们可以让同事通过 导入的方式来使用 ，甚至你可以分享给世界各地的程序员来使用；

如果我们分享给世界上所有的程序员使用，有哪些方式呢？

  • 方式一：上传到GitHub，其他程序员通过GitHub下载我们的代码并手动引用
    • 缺点是大家必须知道你的 GitHub仓库地址 ，并且从GitHub上手动下载
    • 需要在自己的项目中 手动的引用 ，并且 管理相关的依赖
    • 不需要使用的时候，需要 手动来删除相关的依赖
    • 当遇到 版本升级 或者 切换 时，需要重复上面的操作
  • 方式二：使用一个专业的工具来管理我们的代码
    • 我们 通过工具将代码发布到特定的位置
    • 其他程序员直接通过工具来 安装、升级、删除我们的工具代码

显然，通过第二种方式我们可以 更好地管理自己的工具包 ，其他人也可以更好地使用我们的工具包。

# 包管理工具npm

npm：

  • Node Package Manager，也就是 Node包管理器
  • 但是目前已经不仅仅是Node包管理器了，在 前端项目 中我们也在使用它来管理依赖的包
  • 比如vue、vue-router、vuex、express、koa、react、react-dom、axios、babel、webpack等等

如何下载npm工具呢？

  • npm属于node的一个管理工具，所以我们需要先安装Node；
  • node管理工具：node官网 (opens new window)

npm管理的包可以在哪里查看、搜索呢？

  • npm官网 (opens new window)
  • 这是我们安装相关的npm包的官网

npm管理的包存放在哪里呢？

  • 我们发布自己的包其实是发布到 registry 上面的
  • 当我们安装一个包时其实是从 registry 上面下载的包

# npm的配置文件

那么对于一个项目来说，我们如何使用npm来管理这么多包呢？

  • 事实上，我们每一个项目都会有一个对应的 配置文件 ，无论是前端项目(Vue、React)还是后段项目(Node)
  • 这个配置文件会记录着你项目的名称、版本号、项目描述等
  • 也会记录着你项目所依赖的 其它库的信息 和 依赖库的版本号

这个配置文件就是 package.json

那么这个配置文件如何得到呢？

  • 方式一：手动从零创建项目，npm init -y
  • 方式二：通过脚手架创建项目，脚手架会帮助我们生成package.json，并且里面有相关的配置

npm init # 创建时填写相关信息

npm init -y # 创建时使用默认信息

# 配置文件的属性

必须填写的属性： name 、 version

  • name是项目的名称
  • version是当前项目的版本号
  • description是描述信息，很多时候是作为项目的基本描述
  • author是作者相关信息(发布时用到)
  • license是开源协议(发布时用到)

private属性：

  • private属性记录当前的项目是否是私有的
  • 当值为true时，npm是不能发布(publish)它的，这是防止私有项目或模块被粗心发布出去的方式

main属性：

  • 设置程序的入口。
  • webpack虽然会自动找到程序的入口文件，但它默认沿着固定路径寻找，也就是只找程序根目录的index.js文件，
  • 例如当程序的入口文件所在地址为"src/main.js"时，将main属性的值设为该地址即可， "main": "src/main.js"
  • 可以理解为：webpack实际上是找到对应的main属性查找入口文件的

scripts属性：

  • scripts属性用于配置一些脚本命令，以键值对的形式存在
  • 配置后我们可以通过 npm run 命令的 key 来执行这个命令

  • 如果我们将webpack或其他工具安装在dependencies属性上时会有什么后果呢？

# 依赖的版本管理

从package.json中dependencies等存放有库模块和其版本号的属性中，我们会发现其中的版本号例如：^2.0.3或~2.0.3，出现了符号 ^ 和 ~ ，这是什么意思呢？

npm的包通常需要遵从semver版本规范：

  • semver：语义化版本 (opens new window)

semver版本规范是X.Y.Z：

  • X主版本号(major)：当你做了不兼容的API修改(可能不兼容之前的版本)
  • Y次版本号(minor)：当你做了向下兼容的功能性新增(新功能增加，但是兼容之前的版本)
  • Z修订号(patch)：当你做了向下兼容的问题修正(没有新功能，修复了之前版本的bug)

这里解释一下 ^ 和 ~ 的意义：

  • ^x.y.z : 表示x是保持不变的，y和z永远安装最新的版本
  • ~x.y.z : 表示x和y保持不变，z永远安装最新的版本
  • *x.y.z : 表示永远安装最新的版本(不推荐使用)

package-lock.json文件

有时候我们会在目录中看见package.json和package-lock.json两个文件，

当NPM修改node_modules树或package.json时，会自动生成package-lock.json。它描述了生成的确切的树，以便后续安装能够生成相同的树，而不管中间依赖项更新如何。

package-lock.json文件的作用简单，类似于一个缓存目录，存有项目先前下载并缓存过的依赖包的相关信息，里面记录着准确的版本号，没有 ^ 和 ~ 等符号，

当使用者在终端输入并执行 npm install 时，会先去与package-lock.json文件中所记录的版本号进行对比，

如果lock文件记录的版本符合默认 npm install 的package.json版本，即在package.json所记录的版本范围内，比如lock文件中记录有2.4.0版本的axios而package.json所记录的axios版本范围是^2.3.0，则在缓存中寻找并安装lock文件记录的准确版本，值得一提的是，如果在缓存中并未寻找到lock所记录的准确版本，则会前往registry仓库下载该版本；

如果不符合，则会去往registry仓库下载符合范围的最新版本并更新lock文件；

这是为了在多人开发时，避免出现各位开发者执行相同的代码时却输出不同结果的情况。

# npm install命令

安装npm包分两种情况：

  • 全局安装(global install) : npm install webpack -g
  • 项目(局部)安装(local install) : npm install webpack

全局安装

  • 全局安装是直接将某个包安装到全局；
  • 我们一般只将具有工具属性的包安装到全局，比如webpack、babel等，安装完之后便可以在所有的项目中使用了；
  • 类似于axios、express、koa等库文件即便全局安装了之后也并不能让我们在所有的项目中使用axios等库；
  • 但是，我们做项目时这些具有工具属性的webpack、babel等依赖包大多数时候也要进行局部安装，因为不同的项目它所依赖的webpack等工具的版本可能不同，全部只使用一个版本的话可能会出现未知错误；
  • 全局版本的webpack等工具一般是用来做测试。

项目(局部)安装

  • 项目安装会在当前目录下产生一个 node_modules 文件夹
  • 局部安装分为开发时依赖和生产时依赖：
# 安装开发和生产依赖

npm i axios

# 开发依赖
npm install webpack --save-dev
npm install webpack -D
npm i webpack -D

# 根据package.json中的依赖包

# 执行npm install

没有指定版本的话，会默认下载最新版本 ，同时修改package.json的版本记录

实测： 当node_modules中axios模块版本为0.26.0，package.json记录版本为^0.26.0，lock文件记录版本为0.26.0，此时最新版本为0.26.1，那么执行 npm install 命令后，会下载最新版本，将package.json中的记录版本更改为 ^0.26.1，同时修改lock文件记录版本为0.26.1。

( npm会去寻找目录中的 package.json 文件和 package-lock.json 文件，分别获取文件中的 dependencies 属性值和 devDependencies 属性值，这两个属性值都分别存有 依赖库模块名称 和 对应版本号 。)

执行结果需要分情况来说，即目录中是否有package-lock.json文件？

  • 如果检测到目录中没有package-lock.json文件：

    • 分析依赖关系，这是因为我们下载的模块(包)可能会依赖其他的模块(包)，并且多个模块(包)之间可能也会产生相互依赖的情况
    • 从registry仓库中下载对应的代码压缩包(如果我们设置了镜像，那么会从镜像服务器下载压缩包)
    • 获取到压缩包后对该压缩包进行缓存(从npm5开始有的)
    • 将压缩包解压并放进项目的node_modules文件夹中

  • 如果检测到目录中有package-lock.json文件：

    检测package-lock.json文件中各个依赖库模块(包)的版本号是否和package.json中指定范围一致(会按照semver版本规范检测)

    • 不一致，那么会重新构建依赖关系，下载安装符合package.json文件记录范围的最新版本并更新lock文件

    • 一致的情况下，会优先查找缓存中是否有对应的代码压缩包

      • 有，则会获取缓存中的压缩包并解压放进node_modules

      • 没有，则会从registry库下载对应的压缩包进行缓存然后解压放进node_modules

package-lock.json文件解析

  • name：项目的名称
  • version：项目的版本
  • lockfileVersion：lock文件的版本
  • requires：使用requires来跟踪模块的依赖关系
  • dependencies：项目的依赖
    • 当前项目依赖axios，但是axios依赖follow-redireacts
    • axios中的属性如下：
      • version表示实际安装的axios的版本
      • resolved用来记录下载的地址，registry仓库中的位置
      • requires记录当前模块的依赖
      • integrity用来从缓存中获取索引，再通过索引去获取压缩包文件

# yarn工具

另一个node包管理工具yarn：

  • yarn是由Facebook、Google、Exponent 和 Tilde 联合推出的一个JS包管理工具
  • yarn是为了弥补早期npm的一些缺陷而出现的
  • 早期的npm存在很多的缺陷，比如安装依赖速度很慢、版本依赖混乱等等一系列的问题

# cnpm工具

由于一些特殊的原因，某些情况下我们没办法很好的从 https://registry.npmjs.org 下载下来一些需要的包。

查看npm镜像：

npm config get registry
1

我们可以直接设置npm的镜像

npm config set registry https://registry.npm.taobao.org
1

但是对于大多数人来说，并不希望将npm镜像修改了：

  • 第一，不太希望随意修改npm原本从官方下载包的渠道
  • 第二，担心某天淘宝的镜像出现问题或者不维护了，又要重新修改

这个时候，我们就可以使用cnpm，并且将cnpm设置为淘宝的镜像：

npm install -g cnpm --registry=https://registry.npm.taobao.org
1

# npx工具

npx是npm5.2之后自带的一个命令

  • npx的作用非常多，但是比较常见的是使用它来调用项目中的某个模块的命令

  • 我们以webpack为例：

    • 全局安装的是webpack5.1.3
    • 项目安装的是webpack3.6.0

  • 如果我在终端执行 webpack --version 使用的是哪一个命令呢？

    • 显示结果会是 webpack 5.1.3，事实上使用的是全局的，为什么呢？
    • 原因非常简单，在当前目录下找不到webpack时(不会前往node_modules/.bin下查找)，就会去全局找，并且执行命令

  • 那么如何使用项目(局部)的webpack呢？常见的有两种方式：

    • 方式一：明确查找到node_modules下面的webpack
    • 方式二：在scripts定义脚本，来执行webpack

  • 方式一：在终端中使用如下命令(在项目根目录下)

    ./node_modules/.bin/webpack --version
    1

  • 方式二：修改package.json中的scripts

    // package.json
"scripts": {
    "webpack": "webpack --version"
}
    1
    2
    3
    4

  • 方式三：使用npx

    npx webpack --version
    1

npx的原理非常简单，它会到当前目录的node_modules/.bin目录下查找对应的命令

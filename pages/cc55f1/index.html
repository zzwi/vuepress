# 浏览器原理-v8引擎-js执行原理

# 浏览器内核

别称：

  • 排版引擎(layout engine)
  • 浏览器引擎(brower engine)
  • 页面渲染引擎(rendering engine)

不同的浏览器有不同的内核，简单介绍两种（现有内核数量当然不止两种）

  • Webkit：是苹果基于KHTML开发、开源的，用于Safari；
  • Blink：是Webkit的一个分支，Google开发，目前应用于Google Chrome、Edge、Opera等；

​ 浏览器内核包含js引擎

js引擎也有多种，最常用也是这里介绍的便是v8引擎。

# 高级语言

JavaScript是一门高级语言，高级语言同时也分为编译型语言和解释型语言，JavaScript属于解释型语言。

高级语言无法被机器直接执行，因为机器并不认识高级语言，那么高级语言是如何在机器上运行的呢？这其中便有一个转换过程。

# JavaScript引擎(JS引擎)

JavaScript作为一门高级语言，能在机器上执行正是借助了JS引擎。

事实上，我们编写的JavaScript最终都是由机器的CPU执行的，

但是只有机器语言才能被CPU所执行，

而JS引擎便能将我们编写的JavaScript代码编译成机器语言(CPU指令)让CPU执行。

不止是编译，JavaScript引擎的工作包括了编译、执行代码，分配内存和垃圾回收。

比较常见的JavaScript引擎：

  • JavaScriptCore：WebKit中的JavaScript引擎，苹果公司开发；
  • V8：Google开发的超强JavaScript引擎，使Chrome在众多浏览器中脱颖而出

浏览器内核和JS引擎的关系：

浏览器的内核是指支持浏览器运行的最核心的程序，分为两个部分，一个是渲染引擎，另一个是JS引擎;

这里我们以WebKit为例，WebKit事实上由两部分组成：

  • WebCore: 负责HTML解析、布局、渲染等等相关的工作，
  • JavaScriptCore: 解析、执行JavaScript代码;

# V8引擎

  • V8是用C++编写的Google开源高性能JavaScript和WebAssembly引擎，它用于Chrome和Node.js等。
  • V8可以独立运行，也可以嵌入到任何C++应用程序中。

接下来对上图进行简单的解释

Parse 模块会将JavaScript代码转换成AST(抽象语法树),这是因为解释器Ignition 并不认识JavaScript代码;

  • 如果函数没有被调用，那么是不会被转换成AST的；
  • Parse的V8官方文档 (opens new window)

AST (抽象语法树)用于描述程序的结构;

Ignition 是一个解释器，会将AST转换成byteCode字节码

  • 同时会收集 TurboFan 优化所需要的信息(比如函数参数的类型信息，有了类型才能进行真实的运算)
  • 如果函数只调用一次，Ignition 会直接执行将其解释为byteCode
  • Ignition的v8官方文档 (opens new window)

TurboFan 是一个编译器，可以将字节码编译为CPU可以直接执行的机器码;

  • 如果一个函数被多次调用,那么就会被标记为 热点函数 ,然后被TurboFan转换成优化的机器码,提高代码的执行性能;
  • 但是，机器码实际上也会被还原为byteCode，这是因为如果后续执行函数的过程中，类型发生了变化(比如sum函数原来执行的是number类型，后来执行的变成了string类型),所以之前优化的机器码并不能正确的处理运算，就会逆向的转换成字节码;
  • TurboFan的V8官方文档 (opens new window)

V8引擎的解析图(官网例图)

如上图所示，当V8拿到JavaScript代码后的流程如下

  • Blink(浏览器内核)将源码交给V8引擎，Stream获取到源码并进行编码转换；
  • Scanner会进行词法分析(lexical analysis),词法分析会将代码转换成tokens;
  • tokens会被Parser(解析器)转换处理成为AST(抽象语法树),之后再经过Ignition解释器转换为byteCode字节码;
  • tokens会分别经过 Parser 和 PreParser,这是因为解析器在解析代码时需要时间，所以JavaScript引擎都会尽可能避免完全解析源代码，Lazy Parsing (别称：懒解析、惰性解析、延迟解析)便由此诞生了，解析器不必再为每个函数都生成AST,而是可以决定"预解析"(PreParser)或"完全解析"它所遇到的函数;
  • 因为V8实现了Lazy Parsing,所以不会在一开始就解析所有JavaScript代码，将不必要的函数进行 预解析(PreParser) ，也就是只解析暂时需要的内容，而对函数的 完全解析(Parser) 是在函数被调用时才会执行，提高了网页的运行效率

PS :

​ 为了节省不必要的CPU和内存开销，所有主流浏览器都实现了Lazy Parsing。

​ 预解析会检查源代码的语法并抛出语法错误，但不会解析函数中变量的作用域或生成AST。完全解析则将分析函数体并生成源代码对应的AST数据结构。

# JS执行原理

JS引擎在执行代码之前，会在堆内存中创建一个全局对象：Global Object(GO)

  • 该对象所有的作用域都可以访问;
  • 对象里面会包含有Date、Array、String、Number、setTimeout、setInterva 等等;
  • 其中还有一个window属性指向该对象自己;

JS引擎内部有一个 执行上下文栈 (Execution Context Stack,简称ECS),它是用于执行代码的 调用栈 。

什么是 执行上下文栈 ？先上代码

// JavaScript
	console.log(count);// undefined
    count = 2;
	console.log(count);// 2
    var count;

    sum(1,2);// 3
    function sum(num1,num2){
        console.log(num1+num2);
    }

首先，JavaScript是单线程语言，代码的执行顺序应该是由上而下层层递进的，那么上面的count变量在声明之前先赋了值，后台应该是报错的，但是它执行成功了~

这种现象就是 "变量提升"。

"变量提升"就是把变量(var)和函数的声明"移动"到代码的最前面,也就是程序会首先执行这些声明，这时候如果变量只是被声明而还没有被赋值，那么程序便会给变量设置默认值 - undefined。

而"变量提升"发生时，上述代码的全局对象Global Object如下：

// JavaScript
global_Object = {
    count: undefined,
    sum: 0x00 // 指向函数对象
}

之后当代码执行到count=2和sum函数体时，便会给全局对象中对应的属性进行赋值。

创建执行上下文的三种情况：

  • 全局执行上下文：JS引擎在编译全局代码时，创建全局执行上下文。在当前页面中，全局执行上下文仅有一个。
  • 函数执行上下文：函数在被调用时会创建自己的执行上下文。
  • eval函数执行上下文：执行eval函数时，也会创建一个执行上下文。

在执行上下文创建时，var声明的变量会被先赋值为undefined，如果后面有新的赋值才会对undefined进行覆盖，而ES6新增的let/const未被初始赋值undefined，这便是"变量提升"的由来。

执行上下文栈的自我管理：

栈是一种抽象化数据结构，遵循着先进后出、后进先出的基本规则。

还是上代码：

// JavaScript
	one = () => {
      console.log("one");
      two()
    };
    two = () => {
      console.log("two");
    };
    one();// one
		  // two

上述代码中，执行上下文栈的管理顺序如下

  • 创建全局上下文并压入栈中
  • 调用了one()函数，创建了one()函数的执行上下文并压入栈中
  • 进入one()函数内部，调用了two()函数，创建two()函数的执行上下文并压入栈中
  • two()函数执行完毕，将two()移出执行上下文栈
  • one()函数执行完毕，将one()移出执行上下文栈

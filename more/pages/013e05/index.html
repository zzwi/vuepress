JavaScript内存管理

# JavaScript内存管理

首先，我们要知道的是，不管什么样的编程语言，在代码的执行过程中都是需要给它分配内存的，不同的是某些编程语言需要我们手动管理内存，某些编程语言可以自己在一定程度上自动管理内存。

不管是以什么样的方式来管理内存，内存的管理都会有如下的生命周期：

st=>start: 内存分配
use=>operation: 内存使用
e=>end: 内存回收

st(right)->use(right)->e

# JS的内存分配

当我们声明变量、函数、对象的时候，JavaScript会自动为它们分配内存。

来点代码~

// JavaScript
let target = "excellent";
let obj = {
    name: 'zzwi',
    status: 'noMoney'
};
obj.target = target;
console.log(obj.target);// excellent

function sum(count){
    return function add(num){
        return count + num
    }
}

let add10 = sum(10);
console.log(add10(5));// 15

先说一般结论，

  • JS对于基本数据类型 String,Number,Boolean,Undefiend,Null,Symbol,Bigint 内存的分配会在执行时，直接在栈空间进行分配；
  • JS对于复杂数据类型 Object,Array,Function 内存的分配会在堆内存中开辟一块空间，并且将这块空间的指针返回，存于栈空间中进行赋值引用，因此复杂数据类型又称引用数据类型。

这里面关于基本数据类型的内存分配是有错漏的，补充如下：

对象的属性值变量是存在于堆内存中的。

闭包变量也是存在于堆内存中的。

当然，上面这两句也可以理解为一个意思，毕竟函数也算是特殊的对象。

例如上面所示代码中，ob.target便是存储于obj对象在堆内存所开辟的空间中，sum函数的参数count(10)与其所返回的函数add的参数num(5)也都是存于对应的函数对象空间中。

还有一点需要注意，对于=赋值操作，基本数据类型的数据是直接完整地复制变量值，复杂数据类型则是复制对应的指针(地址)。因此会出现下面这种情况：

// JavaScript
let obj = {
    name: 'zzwi',
    status: 'noMoney'
};
let objA = obj;
objA.status = 'study';
console.log(obj.status);// study

# JS的内存使用

内存的使用实际上是对分配的内存进行读写操作。也就是使用变量、函数和对象。

# JS的内存回收

使用完毕，由JS引擎的垃圾回收机制自动找出那些不再使用的变量、函数和对象，然后释放它们所占用的内存。

在这里最艰难的任务便是找到那些不再使用的内存。

JavaScript会自动管理内存，那为什么我们还要了解甚至深入理解内存管理呢？

垃圾回收的英文是 Garbage Collection ，简称 GC 。

GC是怎么找到不需要使用的内存的呢？是GC的算法。

但是，现如今GC的算法也是存在极限的，即便是最精明的GC算法，也无法应对所有的极端情况，不可避免地会出现 内存泄漏 。

这里又提到了一个新的概念--内存泄漏。 什么是内存泄漏？

未能释放已经不再使用的内存 ,这便是内存泄漏。

如果出现了内存泄漏，那么程序占用的内存会越来越大，最终引起客户端卡顿，甚至无响应。如果我们使用Node.js做后端应用，因为后端程序会长时间运行，如果有内存泄漏，造成的后果会更严重，服务器内存可能会很快就消耗光，应用不能正常运行。

# 常见的GC算法

  • 引用计数法：
    1. 当一个对象有一个引用指向它时，那么这个对象的reference count即引用计数就+1，当一个对象的reference count为0时，这个对象就可以被销毁掉
    2. 但是这个算法有一个很大的弊端，那就是 循环引用 .
    3. 如果两个对象相互引用，尽管它们已经不再使用，引用计数法也不会对它们进行回收，导致内存泄漏。-- 循环引用

  • 标记清除法：

    1. 这个算法是设置一个根对象(root object)，垃圾回收器会定期从这个根开始，找所有从根开始有引用到的对象，对于那些没有引用到的对象，就认为是"不可达"，也即不需要/不可用的对象；
    2. 这个算法可以很好地解决循环引用的问题；

  • JS引擎采用比较广泛的就是标记清除算法，当然类似于V8引擎，为了更好的优化，它在算法的实现细节上也会结合一些其他的算法。

# 常见的内存泄漏场景

有8个，分别为 意外的全局变量， 被遗忘的定时器和回调函数 ， DOM的引用 ，闭包 ， 被遗忘的监听者模式 ， 被遗忘的事件监听器 ，被遗忘的Map ，被遗忘的Set 。

  1. 意外的全局变量： 首先明确一点，全局变量是不会被回收的，因此没有被使用的全局变量便会消耗内存，造成内存泄漏。另外还有一点是，在JavaScript中，即便在局部函数作用域中依旧能够定义全局变量，且可能存储大量数据，在不需要使用时依然能占用大量内存，严重影响程序性能，只有在窗口关闭或者刷新页面时才能够被释放，造成意外的内存泄漏。

  2. 被遗忘的定时器和回调函数： 定时器setInterval必须及时清理，否则可能由于其中引用的变量或者函数都被认为是需要的而不会进行回收，如果内部引用的变量存储了大量数据，可能会引起页面占用内存过高，这样就造成意外的内存泄漏。

  3. DOM的引用： 在编写代码时，我们经常会对DOM进行操作，如果是将DOM的引用保存后再操作，这时对DOM的内存回收就需要谨慎处理。

上代码：

// JavaScript
var elements = {
    button: document.getElementById("button")
};
function doStuff() {
    elements.button.click();
};
function removeButton() {
    document.body.removeChild(elements.button);// 清除DOM树的引用
    elements.button = null;// 清除父元素的引用
}

在上述代码中可以看到，我首先是引用了一个id为 button 的DOM元素并将其保存为变量button，之后函数doStuff给变量button添加了点击事件。此时，这个DOM元素已经有了两个引用，一个在DOM树中，一个在对象elements中。因此，将来如果决定清除这个DOM元素时，需要把两个引用都清除，如上述代码中函数removeButton一般。

清除子元素引用时要记得在父元素中对其进行清除，反之，在清除父元素引用时也要注意处理子元素的指向，如果我们进行了父元素的删除操作，却遗漏了它的子元素没有进行处理，此时子元素与父元素依然是引用关系，这会导致父元素仍待在内存中。

还有DOM树内部引用问题，例如表格和它的子节点 <td> ，假如你的JavaScript代码中保存了表格某一个<td>的引用，将来决定删除整个表格的时候，直觉认为GC会回收除了已保存的<td>以外的其它节点，实际情况并非如此，此<td>是表格的子节点，子元素与父元素是引用关系，由于代码保留了<td>的引用，导致整个表格仍待在内存中。

综上，在保存DOM元素引用的时候，要小心谨慎。

# V8引擎的垃圾回收机制

# V8内存限制

V8只能使用系统的一部分内存，具体来说，在64位系统下，V8最多只能分配1.4G，在32位系统中，最多只能分配0.7G。

为什么V8要给它自己设置可使用内存的上限？

有两个影响因素。一个是JS单线程的执行机制，另一个是JS垃圾回收机制的限制。

  • 因为JS是单线程运行的，所以一旦进入到垃圾回收，那么其它的各种运行逻辑都要暂停；

  • 另外，JS的垃圾回收其实是非常耗时间的操作，V8官方是这样形容的：

    以1.5GB的垃圾回收堆内存为例，V8做一次小的垃圾回收需要50ms以上，做一次非增量式的垃圾回收甚至要1s以上。

      		可见其耗时之久。而在这段垃圾回收的时间段内，如前面所讲，其它的各种运行逻辑都要暂停，这会使应用出现明显卡顿，导致应用性能和响应能力直线下降，用户体验暴跌。因此，V8做了一个简单粗暴的选择，那就是限制堆内存。毕竟在一般情况下，前端也不会遇到需要操作几个G内存这样的场景。

​ 不过凡事总有例外，考虑需要手操几个G内存这样特殊场景谁也无法肯定地说绝对不存在，或是有部分用户觉得V8的限制内存还是太大了，想再调小点的，所以V8官方还是给出了能让用户手动调整限制内存的配置命令:

// 这是调整老生代这部分的内存，单位是MB。

node --max-old-space-size=2048 xxx.js

// 这是调整新生代这部分的内存，单位是KB

node --max-new-space-size=2048 xxx.js

# 新生代和老生代

前面提到了新生代和老生代，那什么是新生代，什么又是老生代呢？

对于垃圾回收，V8采用了分代回收的策略，将堆内存分为两部分进行处理：新生代内存和老生代内存 。

新生代是临时分配的内存，存活时间短，老生代是常驻内存，存活时间长。

# 新生代

新生代的内存默认限制：在64位系统下为32MB，在32位系统下为16MB。

新生代内存中的垃圾回收主要通过 Scavenge 算法进行，具体实现时主要采用了 Cheney 算法。新生代的堆内存被分为多个 Semispace ,每个 Semispace 分为两部分 from 和 to ，只有 from 的空间是使用中的，分配对象空间时，只在 from 中进行分配， to 是闲置的。 from 表示正在使用的内存， to 表示现在闲置的内存，进行垃圾回收时按照如下步骤进行：

1. 找出from中还在使用的对象，即活跃对象
2. 将这些活着的对象全部按顺序复制到to
3. 反转from和to，原先的to变为from，from变为to，这时候from中全部是活跃对象，to全部是死亡对象
4. 对to进行全部回收

可以看到在新生代中活跃对象都被复制保存下来，死亡对象都被留在原地全部回收。

Scavenge算法 的作用是将活跃对象排序整齐地复制保存到 to 内存中，避免产生 内存碎片 。Scavenge算法 虽然只能使用新生代一半的内存，但是它只存放生命周期短的对象，这种对象一般很少，因此时间性能非常优秀。

内存碎片： 在内存中无序且零散的空间分布被称为内存碎片，由于堆内存是连续分配的，这样零碎的空间可能会导致稍微大一点的对象没有办法进行空间分配，所以内存碎片问题是我们必须要解决的。

新生代晋升到老生代：

1. 新生代垃圾回收过程中，一个对象经过多次回收依然存在便会被移动到老生代，这种现象叫做晋升。
2. 在from和to进行反转的过程中，如果to空间的使用量超过了25%，那么from的对象全部晋升到老生代

# 老生代

老生代存放的是生命周期较长的对象，他的结构是一个连续的结构，不像新生代分为两部分。老生代垃圾回收有两种方式，标记清除和标记合并。

  • 第一步，标记清除。主要分成两个阶段，即标记阶段和清除阶段。首先会遍历堆中的所有对象，对它们做上标记，然后对代码环境中 正在使用的对象 以及被 强引用 的对象取消标记，剩下的就是要删除的对象了，在随后的清除阶段对其进行内存回收。但是，在标记清除方法清除掉死亡对象后，内存空间就变成不连续的了，出现了内存碎片问题。
  • 第二步，标记合并。主要任务是解决内存碎片问题。V8的解决方式非常简单粗暴，那就是把存活的对象全部往一端靠拢，这是整个过程中最耗时间的部分。
# 小结

与新生代算法相比，老生代主要操作死亡对象，因为老生代都是生命周期较长的对象，每次回收死亡的比较少；而新生代主要操作的存活对象，因为新生代都是生命周期较短的对象，每次回收存活的较少。这样无论新生代还是老生代，每次回收时都尽可能操作更少的对象，效率就提高了。
